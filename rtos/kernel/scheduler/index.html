RTOS 概念页

RTOS 调度器

调度器是内核选择“哪个就绪任务下一个运行”的决策点。只要系统谈得上时序保证，这个保证就一定会经过调度器这一机制。

是什么

RTOS 调度器是一个内核机制，用于在可运行任务之间进行仲裁，并派发下一个应当执行的任务。在大多数嵌入式 RTOS 设计中，这个决策主要由优先级和就绪状态驱动，而不是单纯追求公平。

为什么存在

调度器存在的原因，是多个独立执行上下文会竞争单个 CPU 或有限核心数。如果没有显式仲裁机制，系统就无法受控地体现任务紧急性，也无法在阻塞后恢复，更无法维持对截止时间敏感的行为。

如何工作

当事件、超时或中断让某个任务变为可运行时，该任务就进入就绪态。随后调度器根据既定策略比较所有就绪任务，选择优先者，并在需要时触发上下文切换。在抢占式系统中，更高优先级任务可以立即迫使这一决策发生。

中断或 API 事件
-> 任务状态转为就绪
-> 就绪结构更新
-> 选择最高资格任务
-> 当前任务失去仲裁时触发上下文切换
-> 在新任务上下文下继续执行

内部结构

典型的调度器内部结构包括任务控制块、就绪队列或优先级位图、延时任务链表，以及负责保存和恢复 CPU 状态的上下文切换代码。

  • TCB 保存优先级、栈指针、状态与等待原因。
  • 就绪队列或优先级位图用于快速选择下一个任务。
  • 定时器结构会在延时到期后把任务重新放回就绪集合。

工程问题

调度问题往往先表现为延迟尖峰，而不是立即崩溃。优先级反转、过长临界区、不必要唤醒、过重 tick 开销，都会在算法看上去很简单的情况下逐步侵蚀截止时间可信度。

调度器在理论上可以正确，但如果上下文切换过于频繁，实际成本仍然可能过高。
优先级设计无法脱离互斥策略和中断边界单独存在。
定时 tick 粒度会同时影响响应性与后台开销。

HRTOS 如何实现

HRTOS 偏好显式且有界的调度路径：静态优先级、明确的就绪状态迁移、受控的异步工作来源，以及派发路径中不存在隐藏分配。目标是让任务选择过程既容易理解，也容易测量，更容易审查。
