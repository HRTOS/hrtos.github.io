RTOS 实时系统基础

实时系统的定义取决于它是否满足时序要求，而不是平均运行速度有多快。本页会把这个区别建立起来，并转化成 RTOS 工程模型。

是什么

实时系统指的是：系统正确性不仅取决于逻辑结果是否正确，还取决于结果是否在规定时间内产生。放到 RTOS 语境下，这通常意味着任务、中断与定时服务必须在已知响应窗口内完成。

硬实时：一旦错过截止时间，就应视为系统失败。
准实时：结果一旦变晚就会失去价值，即使系统本身仍在运行。
软实时：时序依然重要，但偶发超时更多表现为质量下降，而非逻辑失效。

为什么存在

之所以需要这个概念，是因为吞吐量无法单独描述系统安全性、控制质量或截止时间可信度。那些需要采样传感器、驱动执行器、维护通信时隙或执行安全逻辑的嵌入式系统，必须用比“平均性能”更强的语言来表达时序保证。

如何工作

实时行为产生于以下条件同时成立的时候：工作被划分为任务与中断，调度规则决定谁先运行，而最坏执行时间与阻塞行为仍然与截止时间兼容。RTOS 本身不会凭空制造实时保证，它只是提供执行这些保证所需的控制面。

外部事件
-> 中断响应
-> 任务唤醒或状态更新
-> 调度决策
-> 有界执行片段
-> 截止时间被满足或被错过

内部结构

底层结构通常包括任务控制块、就绪队列、等待链表、定时服务以及中断交接逻辑。即使在基础认知层，也必须知道：所有时序承诺都依赖于这些内部结构是否有界且足够清晰。

  • TCB 字段定义任务的优先级、栈上下文与状态。
  • 就绪队列与阻塞队列决定任务是否可运行。
  • 定时器链表或计数器决定休眠任务何时重新具备运行资格。

工程问题

大多数实时失败都来自隐藏阻塞、最坏执行时间低估、中断负载过高，或者会引入额外抖动的资源行为。一个设计在演示中看起来很快，但在压力场景下仍然可能不是实时系统。

延迟尖峰往往来自共享资源，而不只是调度器本身。
平均 CPU 利用率几乎无法证明截止时间安全性。
即便只小幅超出时序预算，也足以让硬实时设计失效。

HRTOS 如何实现

HRTOS 通过显式上界来解释实时行为：静态任务定义、可预测唤醒路径、短小中断区间，以及部署前已知的资源规划。它追求的不是最大灵活性，而是不靠猜测也能推导时序行为的系统。
