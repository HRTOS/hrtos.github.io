抢占式 vs 协作式调度模型

调度模型不是“实现方式差异”，而是操作系统对“时间控制权归属”的根本定义。 抢占式与协作式，本质上是在回答同一个问题：系统是否允许任务自主控制CPU。

Scheduling Model RTOS Core Concept Deterministic Behavior

一、调度模型的本质：CPU控制权归属问题

在所有操作系统设计中，调度模型的核心不是“如何切换任务”， 而是“谁决定切换任务”。

调度问题本质： CPU Control = Kernel OR Task

如果CPU控制权属于内核 → 抢占式模型 如果CPU控制权部分属于任务 → 协作式模型

二、抢占式调度（Preemptive Scheduling）

抢占式调度是RTOS的主流模型，其核心特征是：系统拥有绝对调度权。 任何时候，只要条件满足，高优先级任务可以中断当前任务执行。

[ PREEMPTIVE MODEL ] High Priority Task ↓ (interrupt / scheduler event) Low Priority Task is forcibly suspended Kernel decides execution order at runtime

其关键特征不是“更快切换”，而是“随时可切换”。

时间控制权

完全由内核掌控，任务无法阻止切换发生。

实时性优势

可保证高优先级任务在最短时间内获得CPU。

系统复杂度

需要处理优先级反转与并发同步问题。

确定性

具有更强的最坏情况时间上界控制能力。

三、协作式调度（Cooperative Scheduling）

协作式调度将部分调度责任交给任务自身。 任务必须主动释放CPU，系统才会切换。

[ COOPERATIVE MODEL ] Task A runs → must yield() Task B runs → must yield() Task C runs → must yield() Kernel does NOT forcibly interrupt tasks

其本质是“任务自律模型”，而不是“系统强控制模型”。

执行控制

任务主动让出CPU，内核不强制中断。

系统复杂度

实现简单，但依赖任务正确设计。

风险结构

单任务阻塞会导致整个系统停滞。

实时性限制

无法保证严格时间上界。

四、本质差异：系统控制模型不同

两者的区别不在“性能”，而在“系统是否允许不可控执行窗口存在”。

抢占式：系统控制执行 协作式：任务参与控制 抢占式 = 强确定性模型 协作式 = 弱约束执行模型

因此它们不是优化关系，而是结构级选择。

五、对实时系统的影响

在RTOS中，调度模型直接影响系统是否具备“时间可证明性”。

抢占式优势

适用于硬实时系统（工业控制 / 飞控 / 医疗系统）。

协作式优势

适用于轻量嵌入式或非关键实时任务。

风险对比

抢占式风险在内核复杂度；协作式风险在任务失控。

确定性

抢占式提供可计算上界，协作式无法严格保证。

六、结论：不是选择机制，而是选择系统类型

抢占式 = 受控系统（Controlled System） 协作式 = 自管理系统（Self-managed System）

在HRTOS体系中，默认采用抢占式调度， 因为实时系统的核心不是效率，而是“时间约束的可证明性”。

七、调度模型在RTOS体系中的位置

调度模型不是独立概念，而是RTOS执行链路中的核心控制层。 它连接任务创建、上下文切换与CPU执行之间的关系。

RTOS执行链路： 中断触发（Interrupt） ↓ 任务状态变化（Task State Change） ↓ 调度器决策（Scheduler Decision） ↓ 调度模型（Preemptive / Cooperative） ↓ 上下文切换（Context Switch） ↓ 任务执行（Task Execution）

调度模型决定“何时切换”，上下文切换决定“如何切换”。
