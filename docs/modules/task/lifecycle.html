任务生命周期（Task Lifecycle）

任务生命周期定义任务在系统中的状态流转，是调度、同步与资源管理的基础模型。

概述

HRTOS任务生命周期由 OS_TCB.state 驱动，配合调度器与同步模块完成状态迁移。

系统任务状态包括 READY / RUNNING / WAIT / SUSPEND / DEAD， 其中 WAIT 由资源或时间事件触发。

生命周期的本质是：任务在 CPU 执行权与系统资源之间的动态切换过程。

状态模型

HRTOS采用简化状态机模型，所有任务必须处于以下状态之一：

READY // 就绪态 RUNNING // 运行态 WAIT // 等待态（资源/延时/事件） SUSPEND // 挂起态 DEAD // 删除态
WAIT态由 wait_type 决定具体阻塞原因（mutex / sem / msg / event）

状态转换机制

状态转换由调度器、ISR与同步对象共同驱动。

READY → RUNNING：调度器选择最高优先级任务执行（cur_prio）

RUNNING → WAIT：任务调用 os_mutex_lock / os_sem_wait / delay

WAIT → READY：资源释放 / ISR唤醒 / 超时触发

RUNNING → SUSPEND：os_task_suspend()主动挂起

任意状态 → DEAD：os_task_delete()或任务退出

运行流程

CREATE ↓ READY ↓ SCHEDULER SELECT ↓ RUNNING ↓ WAIT / SUSPEND ↓ READY ↓ DEAD

调度触发点包括：tick中断 / ISR退出 / 资源释放 / 优先级变化。

关键结构（基于 HRTOS_CONFIG）

typedef struct { u8 id; u8 base_prio; u8 cur_prio; u8 state; u8 wait_type; u8 wait_flag; u8 wait_obj; u16 wait_tick; } OS_TCB;
cur_prio 可能被 mutex 优先级继承临时提升

任务控制接口（task.h）

os_task_suspend(id) os_task_resume(id) os_task_delete(id) os_task_yield() os_task_exit()

suspend/resume 控制 SUSPEND ↔ READY， delete 将任务置为 DEAD 并回收资源。

等待与资源模型

WAIT状态由 OS_RESOURCE 驱动统一管理。

typedef struct { u8 value; u8 owner; u8 wait_cnt; u16 wait_mask; u8 pending_signal; } OS_RESOURCE;

wait_mask 用位图记录等待任务，支持 O(1) 唤醒。

生命周期特性

  • 状态切换原子化（critical section保护）
  • WAIT统一抽象（delay / sem / mutex / msg / event）
  • 调度驱动状态流转（priority + ready queue）
  • ISR可直接触发READY唤醒
生命周期的核心目标：保证任务状态始终可预测、可追踪、可恢复

常见问题

WAIT 和 SUSPEND 有什么区别？

WAIT是资源驱动阻塞，SUSPEND是主动暂停，两者恢复机制不同。

任务为什么会卡住？

通常是 wait_type 对应资源未释放或事件未触发。

DEAD状态还能恢复吗？

不能，DEAD表示任务生命周期结束，资源已回收。
