HRTOS Task

任务状态模型（Task State Model）

任务状态模型基于TCB与调度器实现，用于描述任务在运行、就绪与等待之间的切换逻辑。

概述

在HRTOS中，任务状态由OS_TCB.state字段维护， 并由调度器、中断与同步对象共同驱动。

状态定义（基于实现）

#define READY 0 // 就绪 #define WAIT 1 // 等待（资源/事件） #define RUNNING 2 // 运行 #define SUSPEND 3 // 挂起 #define DEAD 4 // 删除 这些状态直接对应OS_TCB.state字段，不存在额外“虚拟状态层”。 WAIT不是单一阻塞，而是统一资源等待态（信号量 / mutex / msg / event）。

状态判定机制

核心接口：os_task_get_state() 0 = DEAD（未激活 / 已删除） 1 = RUNNING（当前执行任务） 2 = READY（就绪队列中） 3 = SUSPEND（显式挂起） 4 = PREEMPTED（最近被切出任务） 实际逻辑来源于三类判断： 1）OS_PROCESS_OK：任务是否存在 2）OS_CURRENT_TASK / OS_PREV_TASK：运行态追踪 3）资源位图 / 快速任务标志：判断等待或挂起

状态转换模型

READY → RUNNING (调度器选择) RUNNING → READY (时间片 / 抢占) RUNNING → WAIT (资源等待) WAIT → READY (资源释放/中断唤醒) RUNNING → SUSPEND (主动挂起) ANY → DEAD (删除) 所有转换必须在临界区内完成，避免调度竞争与状态错乱。

等待体系（WAIT扩展）

WAIT_DELAY WAIT_SEM WAIT_MUTEX WAIT_MSG_SEND WAIT_MSG_RECV WAIT_EVENT WAIT_MAILBOX WAIT态不是单一阻塞，而是统一等待模型，由OS_RESOURCE.wait_mask管理。

关键接口

os_task_suspend() os_task_resume() os_task_block() os_task_unblock() os_task_get_state(id) suspend/resume：控制生命周期 block/unblock：资源驱动调度 get_state：状态查询与调试接口

运行流程

CREATE → READY → RUNNING ↓ WAIT / READY ↓ DEAD 状态流转由调度器 + 中断 + 资源系统共同驱动，不存在独立状态线程。

设计说明

  • 状态最小化：减少调度判断成本
  • 资源驱动等待：统一WAIT模型
  • TCB直接控制：避免中间抽象层
  • 中断可恢复性：WAIT → READY可由ISR触发

HRTOS采用“轻状态模型”，避免复杂多状态爆炸， 所有复杂行为由WAIT资源体系实现，而不是扩展状态数量。
