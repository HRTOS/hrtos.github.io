上下文切换（Context Switch）

上下文切换是HRTOS任务调度的核心机制，通过保存当前任务执行现场并恢复目标任务现场， 实现单CPU上的多任务并发执行。

一、内核控制与状态标志体系

HRTOS上下文切换不仅依赖TCB结构，还依赖一组位于BIT / DATA / IDATA / XDATA的底层控制寄存器。

/* 调度控制 */ OS_JINCHENG_CUNCHU -> 优先级筛选开关 OS_SCHED_REASON -> 调度触发类型 OS_RTOS_YES -> 时间片 / 优先级模式 /* 中断控制 */ OS_ISR_FLAG -> ISR干扰屏蔽 OS_INTERRUPT_GAO -> 高优先级中断 os_int_status -> 临界区嵌套控制 /* 高速任务 */ OS_KUAI_PROCESS_A OS_KUAI_PROCESS_B

这些BIT标志构成“软调度控制层”，决定调度器是否进入上下文切换路径。 在中断环境下，OS_ISR_FLAG用于阻断时间片调度，避免现场破坏。

二、上下文保存区（IDATA / STACK）

OS_CURRENT_TASK _at_ 0xE3 OS_PREV_TASK _at_ 0xE4 OS_DISPATCH_ID _at_ 0xE5 os_critical_nesting _at_ 0xE7 OS_INT_BAO[6] -> ACC/B/DPH/DPL/PSW

上下文切换发生时，CPU寄存器首先进入IDATA保护区，再由调度器写入TCB。 其中0xE3~0xE7构成“最小调度状态快照”。

OS_INT_BAO用于中断现场保护，是ISR级上下文切换的第一层保存机制。

三、XDATA任务上下文结构

OS_TASK_CONTEXT[16][...] -> 标准任务上下文 OS_FAST_CONTEXT[2][...] -> 高速任务上下文 OS_JINCHENG_SP[16] -> SP记录 OS_KUAI_SP[2] -> 快速任务SP

XDATA区域是HRTOS的完整任务上下文存储区，负责保存完整执行状态。 标准任务与高速任务采用不同深度的上下文结构以优化切换延迟。

四、上下文切换流程

TRIGGER ↓ os_int_status / OS_ISR_FLAG 判断 ↓ 保存 CURRENT_TASK → IDATA ↓ 写入 OS_TASK_CONTEXT ↓ 调度器选择 NEXT_TASK ↓ 恢复 NEXT_TASK 上下文 ↓ 更新 OS_CURRENT_TASK ↓ RETURN TO EXECUTION

切换路径由调度源决定（时间片 / ISR / 阻塞唤醒）。 所有切换必须在临界区内完成，避免任务状态竞争。

五、设计特点

  • BIT层控制调度路径（极低开销）
  • IDATA实现最小上下文快照
  • XDATA实现完整任务恢复
  • 高速任务与标准任务分层切换
  • ISR与调度器双路径切换模型
