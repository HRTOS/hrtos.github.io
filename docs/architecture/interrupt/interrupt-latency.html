中断延迟（Interrupt Latency）

中断延迟描述从中断触发到 ISR 实际开始执行之间的时间间隔，是实时系统响应能力的核心指标。

系统架构 实时性分析 中断机制

概述

中断延迟（Interrupt Latency）是指从中断事件发生到 CPU 开始执行中断服务程序（ISR）之间的时间间隔，是衡量实时系统响应能力的重要指标之一。

中断延迟越小且越稳定，系统对外部事件的响应越及时，其实时确定性也越高。因此，在 RTOS 设计中，中断延迟通常被作为核心性能指标进行分析和优化。

在 HRTOS 中，中断延迟与 调度系统、 中断流程 以及 系统执行流模型 共同组成完整的实时执行链路。

典型的中断延迟通常包括：硬件响应时间、当前指令完成时间、中断屏蔽等待时间、上下文保存时间以及 ISR 入口开销。

工作原理

当外部硬件中断产生时，CPU 首先完成当前指令的执行，并检查中断屏蔽状态，确保允许中断响应。随后，系统保存当前任务的上下文（寄存器、程序计数器等）并跳转至对应的中断服务程序（ISR）入口。ISR 执行期间可能触发嵌套中断，或受中断优先级和临界区控制影响。

中断处理完成后，ISR 会通过 os_interrupt_exit() 或类似机制通知内核调度器检查是否需要任务切换，从而保证系统高实时性与可预测性。

中断延迟直接影响系统对外部事件的响应速度。为了最小化延迟，ISR 应尽量短小、快速完成，并避免在 ISR 中执行阻塞操作或长时间临界区操作。可以参考 中断延迟 了解优化策略。

关键接口 / 结构

HRTOS 提供了一组用于中断管理的核心接口，包括临界区控制、中断状态查询以及中断上下文中的内核对象操作。这些接口共同构成 HRTOS 的中断编程模型。

os_interrupt_id() // 获取当前中断编号
os_enter_critical() // 进入临界区（关闭中断）
os_exit_critical() // 退出临界区（恢复中断）
os_in_isr() // 判断当前是否运行在中断上下文中
os_queue_send_from_isr() // 中断中发送邮箱消息
os_event_set_from_isr() // 中断中设置事件标志
os_semaphore_post_from_isr() // 中断中释放信号量
os_msgq_send_from_isr() // 中断中发送消息队列数据
os_schedule_request() // 请求调度器重新调度
os_interrupt_exit() // 完成中断退出

以上接口覆盖了 HRTOS 中断处理过程中常用的功能，包括临界区保护、中断状态判断、任务通信、同步对象操作以及中断退出处理，可满足绝大多数中断服务程序（ISR）的开发需求。

主要接口说明如下：

  • os_interrupt_id()：返回当前正在执行的中断编号。
  • os_enter_critical() / os_exit_critical()：进入和退出临界区，防止共享资源在访问过程中被中断打断。
  • os_in_isr()：判断当前代码是否运行在中断上下文中。
  • os_queue_send_from_isr()：在中断中向邮箱发送数据。
  • os_event_set_from_isr()：在中断中设置事件标志。
  • os_semaphore_post_from_isr()：在中断中释放信号量。
  • os_msgq_send_from_isr()：在中断中向消息队列发送数据。
  • os_schedule_request()：请求系统在中断退出后进行一次调度检查。
  • os_interrupt_exit()：完成中断退出，并根据系统状态恢复任务运行。

为保证系统实时性，中断服务程序应尽量保持简短，仅完成必要的数据采集、状态更新和事件通知，将耗时处理放到任务上下文中执行。

运行流程

HRTOS 中断的执行流程确保系统对外部事件的高实时响应：

1. 外部或内部事件触发中断信号；
2. CPU 完成当前指令，并检查中断屏蔽状态；
3. 保存当前任务上下文（寄存器、堆栈指针等）；
4. 跳转至对应的中断服务程序（ISR）；
5. 执行中断服务程序，可根据需要处理事件或通知任务；
6. 中断处理完成，调用 os_interrupt_exit()；
7. 内核检查是否需要进行任务调度；
8. 恢复被中断任务或切换到新的就绪任务。

该流程体现了 HRTOS 对任务和中断的确定性管理，确保高优先级任务及时响应。

扩展说明

中断延迟受到硬件平台和软件设计的共同影响。CPU 架构、总线访问、Cache 状态、中断控制器、临界区长度以及中断优先级配置都会影响最终的响应时间。

为了降低中断延迟，通常建议采用以下优化策略：

  • 保持 ISR 简短，将耗时操作放到任务上下文执行。
  • 缩短关闭中断和临界区持续时间。
  • 合理配置中断优先级，避免高优先级中断被低优先级阻塞。
  • 通过测量最坏情况延迟（Worst-Case Interrupt Latency）评估系统实时性能。

对于硬实时系统而言，相比平均延迟，更应关注最坏情况下的延迟上界，因为它直接决定系统是否能够满足实时约束。
