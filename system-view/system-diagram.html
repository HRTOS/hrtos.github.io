System Model（系统模型）

HRTOS 不是模块集合，而是一个时间因果闭环系统。
所有行为必须映射为可界定的时间函数关系。
核心定义：

系统运行可抽象为循环函数：

Event(t) → Decision(t) → State(t) → Execution(t) → Disturbance(t) → Constraint(t) → Event(t+1)

约束原则： - 所有阶段必须 bounded（有界） - 任一不可预测因素都会破坏确定性
① 事件层（Interrupt Layer）
系统唯一输入来源：中断 / Tick / IPC。

作用：生成调度触发点，打断时间连续性。
↓
② 调度层（Scheduler Layer）
时间资源分配中心。

本质：决定 CPU 在某时间片归属。
↓
③ 状态层（Task Lifecycle）
任务状态集合：Ready / Running / Blocked。

本质：可调度空间的时间划分。
↓
④ 执行层（Execution Flow）
CPU 实际执行轨迹。

必须满足 WCET（最坏执行时间）约束。
↓
⑤ 扰动层（Memory & Cache）
时间不确定性来源：Cache miss / 内存访问 / 分配行为。

本质：时间抖动生成器。
↓
⑥ 约束层（Time System）
系统正确性判定层。

规则：超时 = 错误（不是性能问题）。
↺ Loop Back
系统本质总结：

HRTOS 本质不是操作系统，而是：

一个以时间为核心变量的因果闭环函数系统。

它回答的不是“如何执行”，而是：
👉 “为何在相同时间结构下始终得到一致结果”
